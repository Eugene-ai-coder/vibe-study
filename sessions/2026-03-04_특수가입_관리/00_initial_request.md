# 특수가입 관리 통합 개발 요구사항

## 1. 개요 (Overview)
- **목적**: 특수가입 데이터를 관리하는 CRUD 화면 및 백엔드 API 개발
- **개발 범위**: 테이블 1개(특수가입), 화면 1개(특수가입 관리), 메뉴 등록

## 2. 데이터 구조 (Data Models)

### 2.1. 특수가입 (SPECIAL_SUBSCRIPTION)
- **유형/관계**: 신규 테이블. 가입별과금기준ID+유효시작일 복합 PK
- **항목 명세**:
  | 항목명 | 타입/Key | 필수 | 설명 및 기본값 |
  |---|---|---|---|
  | 가입별과금기준ID (SUBS_BILL_STD_ID) | VARCHAR(20), PK | Y | 가입별과금기준 식별자 |
  | 유효시작일 (EFF_STA_DT) | VARCHAR(8), PK | Y | 유효시작일 (YYYYMMDD) |
  | (기타 컬럼은 리서치 단계에서 기존 유사 테이블 참고하여 결정) | | | |

## 3. 화면 및 UI 구성 (Screens & UI)

### 3.1. 특수가입 관리 화면
- **레이아웃**: 상단(조회조건) → 중간상단(목록 그리드) → 중간하단(입력/조회 폼) → 하단(플로팅 액션바)
- **영역별 구성 요소**:
  - **조회 영역**: 검색 조건 입력 및 조회 버튼
  - **목록 영역**: 조회 결과 그리드 (행 클릭 시 하단 폼에 데이터 표시)
  - **입력/조회 영역**: 선택된 항목의 상세 정보 표시 및 신규/수정 입력
  - **액션바**: 신규, 저장, 삭제 등 버튼

## 4. 이벤트 및 비즈니스 로직 (Events & Business Logic)

### 4.1. 특수가입 관리 화면 관련 로직
- **조회**: 조회조건 입력 후 검색 실행 → 목록 그리드 갱신
- **행 선택**: 그리드 행 클릭 → 하단 입력/조회 폼에 데이터 바인딩
- **신규**: 입력폼 초기화 → 데이터 입력 → 저장
- **저장**: 유효성 검사 후 API 호출 (신규/수정 구분)
- **삭제**: 선택 항목 삭제 확인 후 API 호출

## 5. 기타 요구사항 (Misc & Non-Functional)
- 기존 프로젝트의 가입(Subscription), 가입별과금기준(BillStd) 등 유사 화면의 패턴을 그대로 따를 것
- 사이드바 메뉴에 "특수가입" 항목 등록
